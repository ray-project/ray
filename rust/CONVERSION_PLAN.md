# Ray C++ → Rust Backend Conversion: Complete Plan

## Guiding Principle

**Behavioral identity with the C++ implementation is the supreme goal.** Every phase
includes a "verification gate" that must pass before moving on.  The Rust backend
must produce byte-identical protobuf messages, identical scheduling decisions,
identical error codes, and identical observable behavior from the Python API.

---

## Verification Strategy (applies to EVERY phase)

### 1. Three-Layer Testing Pyramid

```
                    ┌──────────────────┐
                    │  Python E2E      │  ← existing 323 pytest files
                    │  (behavioral)    │     run against Rust backend
                    ├──────────────────┤
                    │  Cross-Language   │  ← NEW: C++ ↔ Rust interop
                    │  Conformance      │     tests that compare
                    │  Tests            │     outputs byte-for-byte
                    ├──────────────────┤
                    │  Rust Unit &      │  ← existing 199 + new tests
                    │  Integration      │     ported from C++ gtest
                    └──────────────────┘
```

### 2. Cross-Language Conformance Test Harness (NEW — build in Phase 0)

A dedicated crate `ray-conformance-tests` that:

- **Runs the same logical test in both C++ and Rust**, captures outputs, and
  asserts byte-equality.
- Uses a simple JSON/protobuf-binary fixture format:
  ```
  fixtures/
    id_generation/
      input.json          ← { "job_id_int": 42, "actor_index": 7 }
      expected_output.bin ← golden binary (generated by C++ once)
    scheduling/
      input.json          ← resource request + cluster state
      expected_output.json← selected node, allocated resources
    rpc/
      request.binpb       ← serialized protobuf request
      response.binpb      ← expected serialized response
  ```
- C++ side: a small binary (`ray_conformance_gen`) that reads `input.json`,
  exercises the C++ code, writes `expected_output.*`.
- Rust side: reads the same `input.json`, exercises the Rust code, compares
  against `expected_output.*`.
- Initially generates golden files from C++, then Rust tests assert match.

**Categories of conformance tests:**

| Category | What is compared | Example |
|---|---|---|
| ID arithmetic | Byte representation of IDs | `ActorID::Of(job_id, ...)` |
| Config parsing | Parsed config struct fields | JSON → RayConfig → field values |
| Protobuf serialization | Wire bytes | Serialize a `TaskSpec`, compare bytes |
| Scheduling decisions | Node selection, resource allocation | Given cluster state + request → decision |
| Resource arithmetic | Fixed-point math results | `FixedPoint(1.5) + FixedPoint(0.5)` |
| Status/error mapping | Error codes, messages | `Status::IOError("x")` → code + message |
| Reference counting | Ref count state after operations | Sequence of AddRef/RemoveRef → final state |
| Pub/Sub message routing | Which subscribers get which messages | Publish to channel → delivered messages |
| Object eviction order | Eviction sequence | Given objects + access pattern → eviction order |
| Placement group packing | Bundle → node assignments | Given bundles + nodes → placement |

### 3. Shadow Mode Testing (Phase 8+)

Before cutting over any component:
- Run both C++ and Rust implementations **side by side** in a real cluster.
- A "shadow proxy" intercepts gRPC traffic, fans out to both, compares responses.
- Log any divergences as P0 bugs — fix before proceeding.

### 4. Python E2E Gate

The existing `python/ray/tests/` suite (323 files) must pass with the Rust
backend before any component is considered done.  This is a hard gate:

```bash
# Phase-specific: run subset relevant to the component
RAY_USE_RUST_GCS=1 pytest python/ray/tests/test_gcs_*.py
RAY_USE_RUST_RAYLET=1 pytest python/ray/tests/test_scheduling*.py
RAY_USE_RUST_BACKEND=1 pytest python/ray/tests/  # final full gate
```

| Risk | Mitigation |
