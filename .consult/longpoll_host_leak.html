1 · Background: what LongPollHost is for

Ray Serve has one controller actor that owns the authoritative state of every deployment — target replica counts, deployment configs, route table, logging config, proxy target groups, fallback targets. Downstream components (each proxy, each router embedded inside a deployment handle, each replica that routes to other deployments) need a coherent view of that state.

A pure push model doesn't scale in actor-RPC terms, so Ray Serve uses long polling:

  • The consumer calls host_actor.listen_for_change.remote(keys_to_snapshot_ids), passing the last snapshot_id it saw for each key it cares about.
  • If any key's current id differs from what the client sent, the host returns immediately with the fresh values.
  • Otherwise the host parks the RPC until notify_changed fires for one of those keys, or a 30–60 s timeout expires (long_poll.py:41-44 — the random range exists specifically to avoid a thundering herd on mass reconnect).

2 · The three dicts and the versioning scheme

The constructor at long_poll.py:298-307:

self.snapshot_ids:       Dict[KeyType, int]   = {}
self.object_snapshots:   Dict[KeyType, Any]   = {}
self._notify_timestamps: Dict[KeyType, float] = {}

notify_changed (long_poll.py:561-593) is the sole writer. The versioning is intentionally opaque to clients:

try:
    self.snapshot_ids[object_key] += 1
except KeyError:
    # Initial snapshot id is random in [0, 1_000_000] — see
    # https://github.com/ray-project/ray/pull/45881 — so clients
    # can safely send -1 to force an initial update.
    self.snapshot_ids[object_key] = random.randint(0, 1_000_000)
self.object_snapshots[object_key]   = updated_object
self._notify_timestamps[object_key] = notify_time

KeyType = Union[str, LongPollNamespace, Tuple[LongPollNamespace, str]] (long_poll.py:74). So a key is one of three shapes: a bare string, a bare namespace enum, or a (namespace, id) tuple. Nothing in the data model bakes in any TTL, reference count, or lifecycle.

The core defect

LongPollHost exposes no remove, pop, clear, evict, forget, or delete API. Grepping def (remove|clear|forget|evict|pop|delete) on the file returns zero matches. The only pops that happen anywhere in the class touch notifier_events (the asyncio-event set for currently-waiting clients), never the three snapshot dicts.

The first four keys are global singletons — the same key is overwritten on every change, so those entries can't grow. The per-deployment keys are the leaky ones. Every live deployment contributes exactly three entries to the three dicts once it has been broadcast at least once.

Gotcha · Java-compat key

The string-keyed (DEPLOYMENT_TARGETS, deployment_id.name) entry is always written, even if no Java router is subscribed. Any cleanup must remove both the tuple-keyed and the string-keyed variants — missing one leaves a zombie entry.

4 · What's actually in each stale value

  • DeploymentTargetInfo (common.py:705-708): is_available: bool + running_replicas: List[RunningReplicaInfo]. Each RunningReplicaInfo is a small frozen dataclass with replica_id, node_id, node_ip, actor_name, max_ongoing_requests, multiplexed_model_ids, routing_stats, port.
  • DeploymentConfig: the user-provided config (autoscaling settings, user_config, graceful timeouts, etc.).
  • snapshot_ids[key]: one int.
  • _notify_timestamps[key]: one float.
Correction worth making

RunningReplicaInfo does not hold an ActorHandle field. It holds actor_name: str, and get_actor_handle() (common.py:700-702) calls ray.get_actor(actor_name, namespace=SERVE_NAMESPACE) lazily. So stale entries do not pin actors alive. This matters for severity: the leak is pure Python-object memory, not actor retention.

5 · The deletion path: where cleanup doesn't happen

When a deployment is deleted, DeploymentStateManager.update() reaches its cleanup block at deployment_state.py:5605-5608:

for deployment_id in deleted_ids:
    self._deployment_scheduler.on_deployment_deleted(deployment_id)
    self._autoscaling_state_manager.deregister_deployment(deployment_id)
    del self._deployment_states[deployment_id]

Three sibling state containers get cleaned — scheduler state, autoscaling state, the manager's own _deployment_states dict. long_poll_host is not one of them. The three per-deployment entries stay in the three dicts for the life of the controller, holding the last-seen DeploymentTargetInfo and DeploymentConfig.

6 · Lifecycle walkthrough

Trace a single DeploymentID("MyDep", "default") through its life:

  1. serve.run(MyDep.bind()) — after the first successful reconcile, broadcast_running_replicas_if_changed and broadcast_deployment_config_if_changed write three entries:
    • object_snapshots[(DEPLOYMENT_TARGETS, DeploymentID("MyDep", "default"))]
    • object_snapshots[(DEPLOYMENT_TARGETS, "MyDep")] (Java)
    • object_snapshots[(DEPLOYMENT_CONFIG, DeploymentID("MyDep", "default"))]
  2. Scale up / down / rolling restart / reconfigure — each event overwrites the same keys. Snapshot ids increment. No growth.
  3. serve.run(MyDep.bind()) again with the same name/app — still the same DeploymentID, still the same keys, still overwrite. No growth.
  4. serve.delete(app='default') — DeploymentStateManager.update() removes self._deployment_states[DeploymentID("MyDep", "default")]. The three long-poll keys survive. They will never be touched again, because no future notify_changed references those keys (the deployment is gone).

A second serve.run with a different app (DeploymentID("MyDep", "app2")) is a new hash and produces three new entries — the previous ones remain. Net: entries accumulate one triple per unique (name, app_name) pair that ever existed in this controller process, never per deploy/redeploy of the same name.

7 · Why this bounds the severity

DeploymentID is @dataclass(frozen=True) with __hash__ over (name, app_name) (common.py:25-39). Hash-identity = overwrite-semantics. So for workloads where deployment names are stable (the normal case: a handful of deployments per app, redeployed many times), the leak is essentially zero. The leak grows only when the identity set grows — e.g. a test suite that creates distinct deployment names for each test, or a long-running controller that sees churn of apps with unique names.

Rough accounting for a 36-test file like test_deployment_actors.py, assuming ~30–50 unique (name, app_name) pairs:

Too small to OOM a controller. In the build-17042 investigation I initially suspected this as the GCS-timeout trigger; the CPU and raylet traces ruled it out and pointed at event-loop starvation via a GetResourceLoad thundering herd instead.

8 · The fix — and the tombstone subtlety

The straightforward fix is an additive API on LongPollHost:

# long_poll.py
def remove_keys(self, keys: Iterable[KeyType]) -> None:
    for k in keys:
        self.snapshot_ids.pop(k, None)
        self.object_snapshots.pop(k, None)
        self._notify_timestamps.pop(k, None)
        # Also wake any still-waiting events so the parked RPC returns.
        for event in self.notifier_events.pop(k, set()):
            event.set()

Called from the deletion loop:

# deployment_state.py — DeploymentStateManager.update()
for deployment_id in deleted_ids:
    self._deployment_scheduler.on_deployment_deleted(deployment_id)
    self._autoscaling_state_manager.deregister_deployment(deployment_id)
    del self._deployment_states[deployment_id]
    self._long_poll_host.remove_keys([
        (LongPollNamespace.DEPLOYMENT_TARGETS, deployment_id),
        (LongPollNamespace.DEPLOYMENT_TARGETS, deployment_id.name),
        (LongPollNamespace.DEPLOYMENT_CONFIG, deployment_id),
    ])

The subtlety lives at long_poll.py:384-391 inside listen_for_change:

try:
    existing_id = self.snapshot_ids[key]
except KeyError:
    # The caller may ask for keys that we don't know about (yet),
    # just ignore them.
    continue

After the key is removed, any client still polling it will silently skip that key forever — no error, no final "deployment gone" delivery.

Why it still works in practice

The controller's normal deletion sequence already does the right thing: the replica state machine transitions all replicas to STOPPED first, which triggers one last broadcast with running_replicas=[] and is_available=False. Clients receive that before the deletion loop runs. Nothing of value is lost as long as the sequence is broadcast-empty → delete state → remove long-poll keys. The existing ordering already puts del self._deployment_states[deployment_id] first, so tacking remove_keys on at the end of the same iteration preserves the ordering.

Belt-and-suspenders alternative

If you want to guarantee a final wake-up for clients whose subscription outlives the deployment, remove_keys can first bump snapshot_ids[key] and broadcast an UNAVAILABLE tombstone value, flush the notifier_events for that key so any parked RPC returns, then pop the three dicts. Belt AND suspenders — at the cost of a small API contract change for what a "removed key" looks like on the wire.
