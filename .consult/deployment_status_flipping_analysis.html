Ray Serve Deployment Status Flipping: DEPLOY_FAILED ↔ UNHEALTHY

1. Problem Statement

We observed an Anyscale Service (Ray 2.54.0, multi-app mode) whose per-deployment status was rapidly flipping between DEPLOY_FAILED and UNHEALTHY during a period of sustained node churn caused by a bad AMI. Specifically:

  • The service was previously healthy for extended periods (days).
  • Worker nodes (and occasionally the head node) were repeatedly dying and being replaced due to an AMI issue.
  • No user-initiated config update or rollout was performed during this period.
  • The Anyscale polling agent on the head node queries /api/serve/applications/ every 1 second.
  • The agent observed the per-deployment (status, status_trigger) pairs oscillating between:
    • (UNHEALTHY, HEALTH_CHECK_FAILED)
    • (DEPLOY_FAILED, HEALTH_CHECK_FAILED)
  • Head node change count: 7 times over 4 hours.
  • The flipping frequency appeared much higher than the head node change frequency, initially suggesting another mechanism was in play.

2. The Deployment Status State Machine

The key insight is that the Ray Serve deployment status state machine has no direct edge from UNHEALTHY to DEPLOY_FAILED. It must pass through UPDATING, and UPDATING can only be entered via a CONFIG_UPDATE trigger (or via a fresh DeploymentState constructor call).

2.2. The Only Valid Oscillation Cycle

For the status to flip between DEPLOY_FAILED and UNHEALTHY, the full path through the state machine must be:

    DF[DEPLOY_FAILED]:::failed
    H1[HEALTHY]:::healthy
    UH[UNHEALTHY]:::unhealthy
    UP[UPDATING]:::updating
    DF2[DEPLOY_FAILED]:::failed

    DF -->|all replicas recover| H1
    H1 -->|HEALTH_CHECK_FAILED| UH
    UH -->|CONFIG_UPDATE| UP
    UP -->|HEALTH_CHECK_FAILED| DF2

At 1-second polling, the transient HEALTHY and UPDATING states (which may last only a single control-loop iteration, ~100ms) can be missed. The polling agent effectively sees only the two "stable" states: DEPLOY_FAILED and UNHEALTHY.

Key constraint: The cycle cannot sustain itself without a CONFIG_UPDATE trigger. From UNHEALTHY, every failure trigger (HEALTH_CHECK_FAILED, REPLICA_STARTUP_FAILED) loops back to UNHEALTHY. The deployment is trapped in UNHEALTHY ↔ HEALTHY unless CONFIG_UPDATE fires.

How these five pieces fit together: Constructor (1) boots the deployment into UPDATING. Activity Probe retries invoke deploy() (2) which re-enters UPDATING via CONFIG_UPDATE. Node churn triggers _stop_replica_mark_unhealthy_if_target_version (5), producing HEALTH_CHECK_FAILED. While in UPDATING, that trigger lands in DEPLOY_FAILED (3). Once the deployment finally reaches HEALTHY and then degrades, the trigger lands in UNHEALTHY (4), and the state machine will not let it return to DEPLOY_FAILED without another CONFIG_UPDATE.

4. Product-Side Context: Activity Probe Retry Behavior

On the Anyscale product side, the Activity Probe on the head node re-applies the Serve configuration when the controller or the head node restarts. This is not a user-initiated rollout — it is part of the automated recovery process.

The Activity Probe performs this configuration application with 3 retries. On each controller/head-node restart, the config is submitted up to 3 times in succession. Each submission that reaches the controller triggers a full deploy() call for every deployment in the application.

4.1. Impact on the State Machine

Each Activity-Probe retry produces one CONFIG_UPDATE trigger per deployment. This means a single controller/head-node restart can inject up to 3 CONFIG_UPDATE events in rapid succession. Each CONFIG_UPDATE pushes deployments back into UPDATING — opening a fresh window during which target-version replica failures (caused by the ongoing worker-node churn) transition the status to DEPLOY_FAILED.

5. Final Deduction

The observed flipping between DEPLOY_FAILED and UNHEALTHY is caused by the combination of:

  1. Head node / controller restarts during the bad-AMI incident. Each restart creates fresh DeploymentState instances initialised to UPDATING and wipes the in-memory runtime status (only target state is checkpointed).
  2. Activity Probe 3x config-apply retries on each controller/head-node restart. Each retry fires a CONFIG_UPDATE trigger per deployment, creating up to 3 fresh UPDATING windows per restart.
  3. Continuous worker-node churn, which causes target-version replicas to die. A replica death while the deployment is in UPDATING transitions the status to DEPLOY_FAILED; after the deployment finally reaches HEALTHY, the same replica death transitions it to UNHEALTHY instead.

Therefore, for a brief period after each controller/head-node restart — bounded by the Activity Probe retry count and the time for replicas to stabilise at the new target version — the deployment can legitimately be observed flipping through UPDATING → DEPLOY_FAILED → HEALTHY → UNHEALTHY, with the transient HEALTHY/UPDATING states missed at 1-second polling.

Once the Activity Probe retries are exhausted and no further CONFIG_UPDATE fires, the deployment settles into a HEALTHY ↔ UNHEALTHY cycle for the remainder of that controller's lifetime. It cannot re-enter DEPLOY_FAILED without another controller restart (which re-triggers the Activity Probe) or an explicit config resubmission.

In short: the flipping is a real artefact of controller-restart + Activity-Probe-retry + node-churn interacting with a state machine that routes failures to different terminal statuses depending on whether the deployment is currently in UPDATING or HEALTHY. It is brief, bounded, and self-resolving after restart sequences finish.
